# Implementation Plan: Platform Connections (API contract)

## Summary

Add the Goa design for the singleton connection API to the existing campaign service, regenerate the Goa code, and implement stub service methods so the service compiles and serves the contract end-to-end. Persistence is a later feature. The design mirrors committee-service idioms (JWT security, bearer-token payload, `If-Match`/`ETag` headers, standard error set) while keeping the repo's established layout (root `design/`, `server.go`).

## Technical Context

- **Language/Framework**: Go 1.25, Goa v3.25.3 (`make apigen` regenerates `gen/`).
- **Layout**: `design/` at repo root (multiple files allowed in package `design`), `internal/service` implements the generated interface, `cmd/campaign-service/server.go` wires the HTTP server.
- **Auth**: JWT security in the DSL (Heimdall-issued, audience = this service). Gateway (Heimdall + OpenFGA) gates every route on `campaign_manager`; the service validates the JWT in-app.
- **Reference**: `linuxfoundation/lfx-v2-committee-service` `cmd/committee-api/design/`.

## Approach

1. `design/connection.go`: define `JWTAuth` (JWTSecurity), reusable attributes (bearer token, if_match, etag, project_id, provider path params), the standard error types, and per-provider typed payload/result types. Define the six methods (`create`, `get`, `update`, `delete`, `test`, `set-credential`) for each provider under paths `/projects/{projectId}/connection-{provider}`.
   - To avoid ~250 lines Ã— 7 providers of duplication, use Go helper functions that emit the shared method shape parameterised by provider key + payload type, called once per provider.
2. `make apigen` â†’ regenerate `gen/`.
3. `internal/service/connection.go`: implement the generated connection method signatures as stubs returning a not-implemented error, so `make build` and `make test` pass. Real logic lands in LFXV2-2556.
4. `make build` + `make test` + `make lint` green.
