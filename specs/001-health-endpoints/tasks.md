# Tasks: Health Endpoints (Readyz & Livez)

**Input**: Design documents from `/specs/001-health-endpoints/`

**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/

**Tests**: Included — the spec explicitly requires automated tests (FR-010, SC-001).

> **Implementation status (2026-07-01)**: All tasks complete. Local gate green (`make check-fmt lint test build build-release`); both endpoints serve `200`/`text/plain`/`OK\n` and `/readyz` returns `503` when not ready (100% coverage of the service package); both endpoints excluded from generated OpenAPI; CI workflows repaired; chart renders probes to `/livez` and `/readyz`.
>
> **Deviations from the task plan (discovered during implementation):**
> - **T005/T012/T013**: The generated `svc.Service` implementation lives in `internal/service/service.go` (not `cmd/campaign-service/service_endpoint.go`) because this scaffold wires the service via `internal/container` (which cannot import `package main`). `internal/service.CampaignService` implements `Readyz`/`Livez` and `ServiceReady()`; the readiness 503 branch is implemented there. Tests are in `internal/service/service_test.go`.
> - **Pre-existing scaffold gaps fixed to reach a compiling state (FR-014)**: added the missing constants referenced by `internal/infrastructure/config/config.go` to `pkg/constants/constants.go`, and fixed a pre-existing `staticcheck` (SA1029) context-key issue in `internal/middleware/request_id.go` so `make lint` passes.
> - **T020/T021**: Local container build (`make docker-build`) and cluster deploy (`make helm-install-local`) could not be executed in this environment (no running Docker daemon, no reachable cluster). Instead validated: `helm lint` passes and `helm template` renders liveness→`/livez`, readiness→`/readyz`, startup→`/readyz`. These paths are exercised by the repaired CI workflows.

**Organization**: Tasks are grouped by user story. Because the two endpoints are co-generated by Goa from a single `design/` package, the compiling/serving bootstrap (FR-014) is a shared Foundational prerequisite; each story then owns its endpoint's behavior guarantees, tests, and validation.

- **[P]**: Can run in parallel (different files, no dependencies on incomplete tasks)
- **[Story]**: US1 (liveness), US2 (readiness), US3 (build/validate/deploy)

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: none — start immediately.
- **Foundational (Phase 2)**: depends on Setup — BLOCKS all user stories.
- **User Stories (Phase 3–5)**: depend on Foundational. US1 and US2 touch the shared test file (`service_endpoint_test.go`) so run sequentially (US1 → US2). US3 depends on US1+US2 being functional (deploys/validates the endpoints).
- **Polish (Phase 6)**: depends on all desired stories complete.

### User Story Dependencies

- **US1 (P1)**: after Foundational. Independently testable (liveness).
- **US2 (P1)**: after Foundational. Adds the readiness 503 gating + tests; shares `service_endpoint.go`/`service_endpoint_test.go` with US1 (sequential, not parallel).
- **US3 (P2)**: after US1+US2 (needs serving endpoints to validate probes/deploy).

### Within Each User Story

- Write tests, then implement; models/predicate before service methods; re-run `make apigen` after any `design/` change.

- T022 / T023 / T024 run in parallel during Polish.

## Implementation Strategy

### MVP First

1. Phase 1 (Setup) → Phase 2 (Foundational: compiling, serving bootstrap).
2. Phase 3 (US1 liveness) → validate.
3. Phase 4 (US2 readiness) → validate. **Functional MVP** = both P1 endpoints working locally.

1. Setup + Foundational → service compiles and serves.
2. US1 → liveness validated.
3. US2 → readiness validated (functional MVP).
